## 1. 项目背景与问题陈述

### 1.1 背景

软件研发团队普遍使用 Jira、Azure DevOps 等工具管理敏捷项目，这些工具积累了大量结构化（sprint、story point、状态、工期）与非结构化（评论、描述、复盘记录）数据。但现有工具存在明显断层：

- **传统 BI/报表工具**（Jira 自带 Dashboard、Power BI、Tableau）擅长"展示数据"，但不解释数据、不主动识别风险，需要使用者自己具备数据解读能力；
- **通用大语言模型工具**（ChatGPT 等）擅长自然语言交互和文本生成，但缺乏对项目管理领域指标的专业理解，且直接喂给 LLM 原始数据容易产生不准确甚至虚构的结论（hallucination），不可直接用于决策。

### 1.2 待解决的问题

> 项目经理和管理层需要在没有专业数据分析师支持的情况下，快速、可信地从项目运行数据中获得"发生了什么、为什么发生、该怎么办"这三层信息，而不是仅仅看到一堆图表。

### 1.3 项目目标场景（本次收窄范围）

本项目聚焦 **软件研发 / 敏捷项目管理** 场景，以 **Jira 风格数据结构**（Issue、Sprint、Story Point、Status 等字段）作为主要数据输入格式，不做跨行业通用化，以保证系统设计的针对性和可实现性。

---

## 2. 产品愿景与目标

**产品愿景：** 让每一位项目经理都能像拥有一位数据分析师一样，用自然语言就能获得可信、可解释、可执行的项目洞察。

**产品目标（本次毕业设计范围内）：**
1. 构建一个端到端可运行的 Web 平台原型（非纯设计稿），验证"统计分析 + LLM"混合架构在项目管理决策支持场景中的可行性；
2. 证明相较于"直接把数据丢给 LLM"的朴素方案，"结构化风险计算 + LLM 基于事实生成解释"（grounded generation）能显著降低幻觉率、提升可信度；
3. 通过量化实验与用户测试，验证系统在准确性、有效性、可用性三个维度上均达到可接受标准。

## 3. 目标用户

### 3.1 用户群体定义

| 用户类型 | 特征 | 核心诉求 |
|---|---|---|
| 一线项目经理 / Scrum Master | 直接管理 1-3 个敏捷团队，日常需要准备站会/评审材料 | 快速掌握项目健康度，及时发现风险 |
| 部门管理层 / 工程主管 | 管理多个项目经理及其团队，不直接接触原始数据 | 高层摘要、跨项目对比、资源调配建议 |

（本项目不面向企业级多角色权限体系的采购决策者，如 CTO/采购负责人——这属于超出毕业设计范围的 B2B 销售场景。）

### 3.2 用户画像（Personas）

#### Persona 1：Sarah — Scrum Master / 项目经理

| 属性 | 内容 |
|---|---|
| 角色 | 某中型软件公司 Scrum Master，同时支持 2 个 Sprint 团队 |
| 数据能力 | 熟悉 Jira 操作，但不擅长 SQL / Excel 数据透视表 |
| 场景痛点 | 每个 Sprint 复盘前需要手动翻看 Jira，整理速度图、延期任务，耗时且容易遗漏问题 |
| 核心诉求 | "我想在 5 分钟内知道这个 Sprint 有没有异常，以及为什么会这样" |
| 使用场景 | 每周 Sprint Review 前 / Retrospective 前上传最新导出数据，快速查看仪表盘和风险提示 |

#### Persona 2：David — 工程部门主管（Executive Stakeholder）

| 属性 | 内容 |
|---|---|
| 角色 | 工程部门主管，管理 4 位项目经理汇报的多个项目 |
| 数据能力 | 只关心结论，不想看原始图表细节 |
| 场景痛点 | 项目经理的口头汇报主观性强，缺乏统一、可比的项目健康度标准 |
| 核心诉求 | "给我一段能直接放进管理周报的项目摘要和建议，不要让我自己去解读图表" |
| 使用场景 | 每月/每季度查看跨项目的执行摘要和管理建议，用于资源调配决策 |

### 3.3 用户故事（User Stories）

| Epic | 用户故事 | 优先级 |
|---|---|---|
| 数据接入 | 作为 Sarah，我想直接上传 Jira 导出的 CSV/Excel 文件，这样我不需要手动整理格式就能开始分析 | Must |
| 仪表盘 | 作为 Sarah，我想一进入系统就看到本 Sprint 的核心 KPI（速度、燃尽图、缺陷率），这样我能快速掌握整体状态 | Must |
| 风险识别 | 作为 Sarah，我想系统自动标出异常（如速度骤降、任务大量延期），这样我不用逐条检查 Jira 记录 | Must |
| 智能解释 | 作为 Sarah，我想知道"为什么"某个指标异常，而不只是看到一个红色警告，这样我能在站会上给团队解释 | Must |
| 自然语言查询 | 作为 Sarah，我想直接用"这个 Sprint 谁的任务最多被阻塞？"这样的问题查询数据，这样我不需要学习任何查询语法 | Must |
| 执行摘要 | 作为 David，我想获得一段自动生成的项目执行摘要和管理建议，这样我可以直接用于管理层汇报 | Must |
| 报告导出 | 作为 David，我想把摘要和关键图表导出为文档，这样我可以分享给不使用本系统的同事 | Should |
| 追问细节 | 作为 David，我想针对摘要中的某个结论继续追问细节，这样我能在决策前进一步确认依据 | Should |

## 4. 核心价值主张与差异化

| 维度 | Jira 原生 Dashboard | Power BI / Tableau | 通用 LLM 聊天工具（如直接用 ChatGPT 分析导出文件） | 本产品 |
|---|---|---|---|---|
| 可视化能力 | 中等 | 强 | 弱 | 中等（够用，非核心卖点） |
| 异常自动识别 | 无 | 需人工配置规则 | 无固定标准，易漏判 | 有（统计规则+可扩展） |
| 自然语言交互 | 无 | 有限（Q&A 插件，泛化差） | 强，但缺乏事实依据 | 强，且基于计算结果生成（grounded） |
| 结论可信度 | 高（原始数据） | 高 | 低（容易幻觉/编造数字） | 高（先算后写，LLM 只负责解释和组织语言） |
| 面向角色 | 数据本身 | 数据分析师 | 通用场景 | 项目经理与管理层 |

**差异化核心：** 本产品不是"更好看的 Dashboard"，也不是"套壳 ChatGPT"，而是把**统计计算的确定性**和**LLM 的语言表达能力**分开设计、显式衔接——先由确定性算法算出事实和异常，再让 LLM 在事实约束下生成解释和建议，从架构上抑制幻觉。

## 5. 产品目标与成功指标

| 类别 | 指标 | 目标值 |
|---|---|---|
| 核心任务完成 | 用户从上传数据到获得第一条洞察的耗时 | ＜2 分钟 |
| 异常检测有效性 | 在注入合成异常的测试数据上的 F1 分数 | ＞0.75 |
| 自然语言查询 | NL2SQL 查询执行准确率（基于预设问题集） | ＞80% |
| 生成内容可信度 | 摘要内容事实一致性（人工/LLM-judge 评分） | Grounded 版本显著优于朴素 Prompt 基线 |
| 可用性 | System Usability Scale (SUS) 平均得分 | ＞68（行业平均线以上） |
| 任务成功率 | 用户测试中核心任务（上传→查看风险→自然语言提问→查看摘要）独立完成率 | ＞80% |

## 6. 功能范围（MoSCoW 优先级）

### Must Have（核心闭环，决定项目及格线）
- CSV/Excel 数据上传与字段映射（Jira 风格 schema）
- 核心 KPI 自动计算与可视化仪表盘（速度、燃尽图、周期时间、缺陷率等）
- 基于统计规则的异常检测（延期、速度骤降、阻塞堆积等）
- 基于结构化结果的 LLM Grounded 摘要与建议生成
- 自然语言查询（NL2SQL，针对结构化字段）
- 基础的量化评估与用户测试闭环

### Should Have（提升竞争力，决定 First Class 上限）
- RAG 检索：基于 Issue 描述/评论等非结构化文本的问答与上下文补充
- 摘要/图表导出为 PDF/Word
- "朴素 Prompt vs Grounded 方案"对比实验模块（用于论文实验章节，也可作为产品内的可解释性展示）

### Could Have（时间充裕再做）
- 多项目/多 Sprint 对比分析
- 简单趋势预测（如基于历史速度预测下一 Sprint 完成量）
- 针对某条洞察的追问式多轮对话

### Won't Have（明确排除，写入论文 Limitations）
- 企业级多角色权限体系（RBAC）
- 与 Jira/Azure DevOps 的实时 API 集成（仅支持文件导入）
- 移动端原生 App

## 7. 非功能性需求概述

| 类别 | 要求 |
|---|---|
| 性能 | 单文件（约 5,000 条 Issue 内）上传到仪表盘渲染完成 ＜10 秒 |
| 可用性 | 核心操作路径不超过 3 步；无需培训即可完成首次上传与查看 |
| 安全与隐私 | 上传数据仅用于当次会话分析，不永久存储原始敏感字段；调用第三方 LLM API 前进行必要的字段脱敏说明 |
| 可维护性 | 前后端分离，模块化设计，便于后续扩展新的 KPI 或异常规则 |
| 可解释性 | 所有 LLM 生成结论必须可追溯到具体的计算结果或数据片段 |

## 8. 关键假设与风险

| 假设/风险 | 说明 | 应对策略 |
|---|---|---|
| 数据可获得性 | 无法获取真实企业 Jira 数据 | 使用 Kaggle 等公开项目管理数据集 + 自建 Jira 风格合成数据生成器 |
| LLM API 成本与稳定性 | 频繁调用可能产生费用或速率限制 | 设置调用缓存、限制单次分析的 Token 规模、预留降级方案（如离线规则摘要兜底） |
| 用户测试招募难度 | 真实项目经理不易招募到 5-10 人 | 优先招募有实习/项目管理经验的同学或校内相关背景人员，测试前完成低风险伦理自评 |
| 异常检测评估无金标准 | 真实数据无标注异常 | 采用合成注入异常的方式构造有 ground truth 的测试集 |

## 9. 数据来源与伦理考量（Ethics & Data Privacy）

- **数据来源：** 优先使用公开数据集（如 Kaggle 项目管理/Jira 数据集、GitHub 公共仓库 Issue 导出）与自建合成数据生成器，避免使用任何未授权的真实企业数据。
- **用户测试伦理：** 涉及人类参与者的可用性测试需要在开展前完成学院低风险伦理自评（Ethics Self-Assessment），并对参与者进行知情同意告知。
- **第三方 API 隐私：** 若上传数据中包含人名、真实项目名称等信息，系统在发送给 LLM API 前应说明潜在的数据出境/第三方处理风险，MVP 阶段建议默认使用脱敏后的合成数据进行演示。

## 12. 术语表

| 术语 | 说明 |
|---|---|
| Grounded Generation | 让 LLM 基于给定的确定性事实（如已计算好的 KPI/异常）生成文本，而非自由发挥，以降低幻觉 |
| NL2SQL | 将自然语言问题转换为可执行的结构化查询语句 |
| RAG | Retrieval-Augmented Generation，检索增强生成，先检索相关文本片段再交给 LLM 生成回答 |
| SUS | System Usability Scale，标准化的十题可用性评分问卷 |
| MoSCoW | Must/Should/Could/Won't，需求优先级分类法 |
